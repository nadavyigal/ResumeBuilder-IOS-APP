Why  The highest-leverage fix

Tapping Upload used to drop users straight into the iOS Files picker — often an empty "No Recents" dead-end. This is where ~81% drop. We insert an app-level sheet first: it sets expectations (PDF/DOCX, 5 MB), points to where résumés live (iCloud / Downloads / Mail), and — critically — offers two non-file routes so the funnel can never dead-end on "file not found."

Implementation

SwiftUI .sheet / .presentationDetents([.medium,.large]) over a dimmed Home · "Browse Files" launches .fileImporter ([.pdf,.docx]) · location chips are non-tappable hints (or deep-link to UIDocumentPicker directories) · rows = glass list. Motion: sheet springs up; reduced-motion → fade.

Why  Make the wait do work

The moment after a file lands is when doubt creeps in ("did it work?"). Instead of a blank spinner we show a visible read-through — the file confirmed, a scan animating over a mini render, and a 3-step parse checklist ticking. It buys the 1–3s of parse time while building trust that the app actually read the résumé.

Implementation

Auto-advances to E2 when parse completes (no tap). Checklist rows bind to real parse stages (extract → segment → readability); scan line is a decorative matchedGeometry sweep, hidden under reduced-motion. If parse fails → inline recoverable error with "Paste text instead", never a dead end.

Copy

"Extracted your text" · "Detecting sections & dates" · "Checking ATS readability" · "Reading your résumé the way an ATS does…"

⚑ Flags

Needs parser to emit per-stage progress events so the checklist reflects reality, not a fake timer.

Recovery & dead-end prevention Every failure has a way forward

The funnel only holds if failures don't end it. Each error state explains in plain words, never blames, and offers the same escape hatches as the happy path — so a scanned PDF, an oversized file, or a dropped connection becomes a detour, not an exit.

Why  The most common real failure

A huge share of uploads are scanned / image-only PDFs with no text layer — the silent killer of résumé tools. We name it plainly ("scanned image — no selectable text"), tie it to why it matters ("an ATS can't read it either"), and offer the exact same escape hatches as the happy path. The amber tone reads as a heads-up, not an error.

Implementation

Triggered when extracted text length ≈ 0 after parse · primary recovery = paste-text (highest success) · "Try a sample" guarantees forward motion · offer an optional on-device OCR attempt before giving up. Never returns the user to a blank picker.

Copy

"We couldn't read that résumé" · "It looks like a scanned image — there's no selectable text, so an ATS can't read it either." · "Paste the text instead" · "Tip: 'Export as PDF' keeps the text readable."

⚑ Flags

On-device OCR fallback is optional scope. Paste-text path (shared with Upload sheet) must exist.

Why  Protect the work, remove the panic

A drop mid-analysis is the scariest failure — it feels like losing everything you just did. We invert it: "paused at 72%, your résumé and job are saved, nothing's lost," with auto-resume on reconnect plus a manual Retry. The user's inputs survive an app kill, so coming back is effortless.

Implementation

Inputs persisted locally the moment they're entered · analysis job is resumable / idempotent server-side (job id + checkpoint) · NWPathMonitor drives the offline banner and auto-retry. Amber, not red — this is recoverable. Same pattern covers a server timeout.

Copy

⚑ Flags

Needs a resumable analysis job (checkpoint + idempotent re-run) and local persistence of inputs. Backend.
